# 麻将牌谱分析器 — 设计方案

> 状态：方案待用户确认 → 确认后进入 PoC（先打通最难的"拉取牌谱"一步）
> 参考项目：[zyr17/MajsoulPaipuAnalyzer](https://github.com/zyr17/MajsoulPaipuAnalyzer)（Electron+C++，抓全账号牌谱，对标天凤指标，**有封号风险**）

## 一、需求（已与用户确认）

| 维度 | 选择 |
|---|---|
| 规则 | 日麻（立直麻将），四人场 |
| 数据来源 | **雀魂单局牌谱分享链接**（用户手动粘贴，逐局喂入） |
| 分析产出 | ①单局复盘讲解 ②多局累积的长期数据统计 ③对标高水平基准线 |
| 技术栈 | Python |

## 实测结论（2026-05-31）

用 `src/parse_link.py` 实测用户提供的链接：**直接 HTTP 拉取分享链接返回 200，但只是空壳 SPA 页面（30KB HTML，不含 UUID/对局数据）。证实"匿名直提"不可行**，对局数据必须登录后经 WebSocket `fetchGameRecord` 取回。下一步只能在「浏览器抓 token」/「账号密码登录」/「手动导出 json」三选一。

## 二、关键技术现实（务必先认清）

雀魂分享链接形如 `https://game.maj-soul.com/1/?paipu=<UUID>_<seat>`。

- **不能匿名 HTTP GET**。牌谱内容存在雀魂服务器，要通过 WebSocket + protobuf（liqi 协议）调用 `.lq.Lobby.fetchGameRecord` 才能拿到，**该调用需要登录态**（账号密码登录 / 或已抓到的 access token）。
- 因此即便是"单局链接"，抓取那一步仍依赖登录。相比参考项目"批量爬全账号 1000 局"，单局逐个拉取**频率低、风险显著更小**，但**非零**。封号风险由用户知悉并承担。
- 牌谱内容是一串 protobuf 编码的对局动作流（发牌 / 摸切 / 吃碰杠 / 立直 / 和了 / 流局），解码后才能分析。

### 应对策略（降低工作量与风险）

1. **复用成熟轮子**，不重造协议层：
   - **首选**：[Equim-chan/mjai-reviewer](https://github.com/Equim-chan/mjai-reviewer)（原 akochan-reviewer）。喂雀魂/天凤链接，自动拉取并用 AI 引擎（Mortal/akochan）逐手点评。本身就是单局复盘神器，能直接当"单局讲解"引擎。
   - **协议层 Python 库**：liqi protobuf 解析（如 `mahjong_soul_api` 系列），用于自己拉取+解析成标准 JSON。
2. **登录凭据**：需要用户提供雀魂账号/或在浏览器登录后抓到的 token，存本地 `data/config.json`（chmod 600，不入库）。
3. **降频**：单局手动触发，不做循环批量；加随机间隔。

## 三、架构（Python，分层）

```
牌谱分析/
├── fetch/          # 拉取层：链接 → 雀魂 API → 原始 protobuf
│   └── majsoul.py  # liqi 协议封装（或调用 mjai-reviewer）
├── parse/          # 解析层：protobuf → 标准化对局 JSON（一局一文件）
│   └── normalize.py
├── stats/          # 统计层：跨多局聚合指标
│   └── metrics.py
├── review/         # 单局复盘层（可接 mjai-reviewer / Mortal）
│   └── single_game.py
├── report/         # 出报告层：HTML / 终端
│   └── html.py
├── data/           # 本地数据（不入库）
│   ├── config.json # 登录凭据，chmod 600
│   ├── games/      # 解析后的每局 JSON，累积用于长期统计
│   └── baseline.json # 高水平基准线参考值
└── 方案.md / README.md
```

数据流：`粘贴链接 → fetch → parse(存 games/) → 单局走 review 出复盘 / 多局走 stats 出统计 → report 渲染 HTML`

## 四、分析指标体系（对标天凤/雀魂，分批实现）

### 长期统计（多局聚合）
- **顺位**：平均顺位、一/二/三/四位率、连对率（前二率）、避四率
- **和了**：和了率、平均和了打点、平均和了巡目、副露和了率、默听率
- **放铳**：放铳率、平均放铳打点、立直放铳/副露放铳占比
- **立直**：立直率、立直和了率、立直收支、先制立直率、追立率、里宝率、一发率
- **副露**：副露率、副露后和了率/放铳率
- **被动**：被自摸率、炸庄率（坐庄被自摸）
- **流局**：流局率、流局听牌率、流局收支
- **段位**：安定段位（按场次顺位分布估算）

### 单局复盘（逐手）
- 每巡的牌效 / 危险度提示；立直、鸣牌、押引（押 vs 降）时机评价
- 接 mjai-reviewer/Mortal 给出"AI 推荐打法 vs 实际打法"差异和疑似失误点

### 对标基准线
- `baseline.json` 内置高水平参考值（雀魂魂天 / 天凤凤凰桌公开统计），报告里并排对比并标红弱项。

## 五、分阶段计划

- **Phase 0（PoC，最关键）**：打通"一条链接 → 拿到并解析成 JSON"。这一步通不了后面都是空中楼阁。先验证 mjai-reviewer 能否直接用，再决定是否要自写协议层。
- **Phase 1**：单局复盘讲解（接 mjai-reviewer / 自写基础牌效）。
- **Phase 2**：解析结果落 `data/games/`，实现长期统计指标（先做顺位/和了/放铳/立直几个核心）。
- **Phase 3**：基准线对比 + HTML 报告。
- **Phase 4**：补齐副露细分、安定段位等高级指标。
