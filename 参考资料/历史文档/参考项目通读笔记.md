# MajsoulPaipuAnalyzer 参考项目通读笔记

> 本文档是对 [MajsoulPaipuAnalyzer](https://github.com/zyr17/MajsoulPaipuAnalyzer) 仓库全部文件的逐文件技术总结。
> 该工具是一个雀魂(Majsoul)麻将牌谱分析工具，由两部分组成：
> - **MajsoulPaipuCrawler**（Electron + JS）：牌谱爬虫，从雀魂抓取并转换牌谱
> - **PaipuAnalyzer**（C++）：牌谱分析引擎，批量统计 150+ 项指标并生成 HTML 报告

## 1. 项目整体架构

```
┌──────────────────────────────────────────────────────────────┐
│  main.js (Electron 主进程, 1084行)                            │
│  - 创建两个 BrowserWindow (雀魂网页 + SimpleMahjong回放)      │
│  - 管理 IPC 通信、文件系统读写、牌谱下载/转换调度               │
└──────────────┬───────────────────────────┬───────────────────┘
               │ IPC                       │ require
               │                           │
┌──────────────▼───────────┐  ┌────────────▼──────────────────┐
│ browseWindow (注入脚本)    │  │ Node.js 模块                   │
│                           │  │                                │
│ browseinject.js           │  │ analyze.js — protobuf解码/元数据│
│  - GameMgr/app.NetAgent   │  │ config.js — 配置管理            │
│  - fetchGameRecord API    │  │ protobuf.js — protobuf编解码库  │
│                           │  │ wshook.js — WebSocket hook库    │
│ majsoul2tenhou.js         │  │                                │
│  - 雀魂牌谱→天凤格式转换   │  │                                │
└───────────────────────────┘  └────────────────────────────────┘

          ↓ 输出 JSON 牌谱 (data/{source}/{id}/paipus.txt)

┌──────────────────────────────────────────────────────────────┐
│  PaipuAnalyzer (C++ 命令行程序)                                │
│  src/main.cpp → src/analyzer.cpp → src/output.cpp             │
│  - 读取 JSON 牌谱 + config.json 筛选条件                       │
│  - 模拟每局对局过程，收集 150+ 项统计指标                       │
│  - 生成 PaipuAnalyzeResult.html (嵌入 Vue.js 的交互式报告)     │
│  - 同时控制台表格输出                                          │
└──────────────────────────────────────────────────────────────┘
```

**数据流**：雀魂服务器 → `browseinject.js` 调用 API 获取 protobuf → `analyze.js` 解码为 JS 对象 → `majsouldataconvert` 转为中间 JSON 格式 → 写入 `paipus.txt` → C++ PaipuAnalyzer 读取分析 → 生成 HTML 报告。

**技术栈**：
- 前端：Electron, Laya(雀魂引擎), protobuf.js, jQuery, Vue.js
- 后端：C++ (g++/cmake), CJsonObject(git submodule)
- CI：AppVeyor (Win/Linux), GitHub Actions (macOS/Win/Linux)
- 支持服务器：国服、日服、国际服
- **仅支持四人麻将**分析

### 2.2 `doc/result.md` — 分析结果统计规则

定义了分析时的特殊约定（16 条）：

| # | 规则 |
|---|------|
| 1 | **双响**：铳了局数中算铳了两局 |
| 2 | **三响**：流局了不算铳（雀魂中不存在三响） |
| 3 | **双响点数**：铳了 5200+7700 时分开计数，不相加 |
| 4 | **和了收入**：包含本场供托，包括自己的立直棒 |
| 5 | **素点**：扣除本场和供托后的净点数 |
| 6 | **AL(All Last)**：以刚进入 AL 时的数据为基准；北起 AL 4位连庄+西入飞人视为"避四+逆一" |
| 7 | **副露**：分副露等级统计（待实现） |
| 8 | **默听**：门清未立听牌（非立直状态下的门清听牌） |
| 9 | **暗杠不算副露** |
| 10 | **副露和了率**：和了局数中副露局数的占比（非与总副露局数之比） |
| 11 | **中途飞人**：视为 AL 位序不变 |
| 12 | **炸庄**：坐庄时被自摸 |
| 13 | **里宝率和一发率**：在立直和牌的前提下计算概率 |
| 14 | **流局局数**：不包括途中流局和流局满贯 |
| 15 | **流局点数收支**：不包括流局满贯收支 |
| 16 | **安定段位**：详见 stable-rank.md |

### 2.4 `doc/stable-rank.md` — 安定段位计算方法

#### 核心公式

设 N 盘记录，一至三位顺位 pt 分别为 a1, a2, a3，四位顺位 pt 视为 0。各顺位出现 x1, x2, x3, x4 次。每盘马点 pt 为 m1, m2, ..., mN。

**四位每次可失点**：

```
SR = (a1×x1 + a2×x2 + a3×x3 + m1 + m2 + ... + mN) / x4
```

即：**（一至三位总顺位 pt + 所有盘的总马点 pt）÷ 四位次数**

然后根据 SR 值找到对应的安定段位。不同头衔和星级在四位时掉分不同，同一头衔不同星级掉分为等差数列。

#### 有偏性

该公式是**有偏估计**——有限 N 次采样中，各顺位服从多项分布，pt 期望与公式结果不完全相等。

#### 置信区间

1. 蒙特卡洛模拟（1000 次）得到均值和方差
2. N 较大时近似正态分布
3. 置信区间公式：`CI = [X - t(α/2, N-1) × S/√N, X + t(α/2, N-1) × S/√N]`
4. 最终将区间大小**放大 1.4 倍**（经验系数）

#### 房间参数

| 房间 | 东风 roombase | 东风 roomdelta | 南风 roombase | 南风 roomdelta |
|------|:---:|:---:|:---:|:---:|
| 铜(2) | 0 | 10 | 0 | 20 |
| 银(3) | 30 | 10 | 60 | 20 |
| 金(4) | 70 | 10 | 150 | 15 |
| 玉(5) | 100 | 10 | 195 | 15 |

安定段位 = `(avg_pt - roombase) / roomdelta`

## 3. 抓取与协议层

### 3.1 `lib/majsoul/browseinject.js` — 雀魂页面注入脚本（牌谱抓取核心入口）

**运行环境**：注入到 Electron 的 browseWindow 中，运行在雀魂游戏页面上下文，可直接访问 `GameMgr`、`uiscript`、`app.NetAgent` 等全局对象。

#### 关键函数

| 函数 | 功能 |
|------|------|
| `getAccountID()` | 轮询等待 `GameMgr.Inst.account_id` 就绪，通过 IPC 发送 `userid` 事件 |
| `collectallpaipu(event, callback)` | 自动遍历收集所有牌谱元数据。调用 `fetchGameRecordList` API，分页获取各类别（ALL=0, FRIEND=1, RANK=2, MATCH=4, COLLECT=100），每页 30 条，去重后存入 `uiscript.UI_PaiPu.record_map` |
| `get_more(result, start, types, count)` | 分页递归函数，当前类别取完后切换下一类别 |

#### 调用的雀魂 API

全部通过 `app.NetAgent.sendReq2Lobby` 发送：

| API | 参数 | 功能 |
|-----|------|------|
| `Lobby.fetchGameRecordList` | `{start, count, type}` | 分页获取牌谱列表 |
| `Lobby.fetchGameRecord` | `{game_uuid: uuid, client_version_string}` | 获取单个牌谱完整 protobuf 数据 |
| `Lobby.fetchGameRecordsDetail` | `{uuid_list}` | 批量获取牌谱详细元数据 |

#### liqi.json 获取

```javascript
let res = Laya.loader.load("res/proto/liqi.json").getRes("res/proto/liqi.json");
```

通过 Laya 引擎资源加载器获取 protobuf 协议定义文件和 `cfg.desktop.matchmode` 房间映射。

#### IPC 事件

| 事件 | 方向 | 说明 |
|------|------|------|
| `userid` | renderer→main | 发送用户 account_id |
| `collectpaipu` / `collectpaipucallback` | main↔renderer | 收集牌谱列表 |
| `fetchpaipudata` / `fetchpaipudatacallback` | main↔renderer | 下载单个牌谱完整数据 |
| `fetchliqijsoncallback` | renderer→main | 返回 liqi.json 协议定义 |
| `collectallpaipu` | main→renderer | 触发自动收集全部牌谱 |
| `collectmetadata` | main→renderer | 批量查询牌谱元数据 |

#### 登录态/Token 处理

脚本**不直接处理登录态**，依赖游戏页面已登录后的全局对象：
- `GameMgr.Inst.account_id` — 用户 ID
- `app.NetAgent` — 已携带认证 token 的网络代理
- `uiscript.UI_PaiPu.record_map` — 游戏客户端已加载的牌谱记录

### 3.2 `lib/wshook.js` — WebSocket Hook 库

**来源**：基于 skepticfx/wshook，通用的 WebSocket 拦截库。

**原理**：用自定义构造函数替换全局 `WebSocket`，拦截 `send` 和 `message` 事件。

**API**：
- `wsHook.before(data, url, wsObject)` — 发送前拦截，可修改数据
- `wsHook.after(event, url, wsObject)` — 接收后拦截，可修改事件（返回 null 丢弃消息）
- `wsHook.resetHooks()` — 恢复默认

**关键结构 `MutableMessageEvent`**：仿制的 `MessageEvent`，所有属性（`data`, `origin`, `type`, `target` 等）可写，使 hook 回调能修改接收到的数据后再传给游戏原始处理逻辑。

### 3.3 `lib/protobuf.js` — Protobuf 编解码库

**版本**：protobuf.js v6.10.0 的预编译（bundled）版本。

**在项目中的使用**：
```javascript
ProtobufRoot = protobuf.Root.fromJSON(liqijson);
ProtobufRoot.resolveAll();
ProtobufWrapper = ProtobufRoot.lookupType('.lq.Wrapper');
```

`liqi.json` 是雀魂的 protobuf 协议定义文件，包含 `Wrapper`、`ActionPrototype`、`RecordNewRound` 等所有消息类型。

### 3.4 `lib/config.js` — 配置管理模块

| 函数 | 功能 |
|------|------|
| `getAppDataPath()` | 跨平台配置存储路径（macOS: `~/Library/Application Support/MajsoulPaipuAnalyzer`，Win: `%APPDATA%`） |
| `saveconfig(content)` / `loadconfig()` | 配置持久化 |

## 4. 牌谱格式转换

### 4.1 `lib/majsoul/majsoul2tenhou.js` — 雀魂→天凤格式转换

**运行环境**：注入到 browseWindow 中执行的 IIFE。天凤格式是麻将社区通用交换格式，可被 tenhou.net/5 查看器和 Akochan 等分析工具使用。

#### 全局常量

- `TSUMOGIRI = 60` — 天凤中自摸切标记
- `DAISANGEN = 37, DAISUUSHI = 50` — 包牌役种 ID
- `YSCORE = [[0, 16000, 48000], [16000, 8000, 32000]]` — 役满得分表（亲/子/荣和）

#### 牌编码转换 `tm2t(str)`

雀魂用字符串（`"3m"`, `"7z"`），天凤用整数：
- 11-19: 1-9 万（m）
- 21-29: 1-9 筒（p）
- 31-39: 1-9 索（s）
- 41-47: 东南西北白发财
- 51/52/53: 赤5万/筒/索

#### `kyoku` 对象（局信息管理）

```javascript
kyoku.init(leaf)    // 从 RecordNewRound 初始化
kyoku.dump(uras)    // 输出天凤格式
kyoku.countpao(tile, owner, feeder)  // 包牌计数器
```

局信息字段：
- `round: [场风×4+局, 本场, 立直棒数]`
- `initscores` — 初始分数
- `doras` — 宝牌指示牌
- `draws[4]` / `discards[4]` — 四家摸牌/打牌记录
- `haipais[4]` — 四家配牌
- `dealerseat` — 庄家座位
- `nowinds[4]` / `nodrags[4]` — 各家风牌/三元牌碰杠计数（包牌判定用）
- `paowind` / `paodrag` — 包牌责任人

#### 吃碰杠的天凤 naki 编码

| 前缀 | 含义 | 牌数 |
|------|------|------|
| `"c"` | 吃 (chii) | 3张 |
| `"p"` | 碰 (pon) | 3张，`p` 标记在被打出的牌前 |
| `"m"` | 大明杠 (daiminkan) | 4张 |
| `"k"` | 加杠 (shouminkan) | 4张，替换原碰记录中的 `p` |
| `"a"` | 暗杠 (ankan) | 4张 |

相对座位计算 `relativeseating(seat0, seat1)`：0=上家, 1=对面, 2=下家。

#### 和了得分转换 `parsehule(h, kyoku)`

最复杂的部分，处理：
1. 自摸 vs 荣和的分数计算
2. 亲/子不同支付方案
3. 立直棒收取（双荣和时第二人不能收立直棒）
4. 本场支付
5. 包牌（大四喜/大三元）：自摸时责任人返还其他玩家支付；荣和时责任人支付放铳者一半 + 全部本场
6. 三麻自摸损失补偿
7. 满贯/跳满/倍满/三倍满/役满/数え役满/切り上げ满贯判定

#### 天凤日志顶层结构 `parse(record)`

```javascript
{
    ver: "2.3",
    ref: uuid,              // 牌谱 UUID
    log: [...],             // 各局日志
    ratingc: "PF4"/"PF3",   // 4人/3人
    rule: { disp, aka53, aka52, aka51 },
    dan: [...],             // 段位名称
    rate: [...],            // 段位分数
    name: [...],            // 玩家昵称
    sc: [...]               // 分数 (part_point, total_point/1000 交替)
}
```

### 4.2 `lib/majsoul/analyze.js` — 牌谱分析核心模块（Node.js 主进程）

#### Protobuf 消息解码

**初始化** — `AnalyzeInit(liqijson, roomid)`：
```javascript
ProtobufRoot = protobuf.Root.fromJSON(liqijson);
ProtobufWrapper = ProtobufRoot.lookupType('.lq.Wrapper');
```

**解码流程** — `Protobuf2Object(data)`：

| 首字节 | 类型 | 处理 |
|--------|------|------|
| `0x01` | Server Message | 服务端主动推送 |
| `0x02` | Client Request | 提取 2 字节请求 ID（小端序），Wrapper 解码后用 requestType 解码，缓存 `{responseType}` |
| `0x03` | Server Response | 提取请求 ID，从缓存取 responseType 解码 |
| 其他 | 裸数据 | 直接用 Wrapper 解码 |

**递归解码** — `ProtobufDecode(arr8, recursive, ProtobufType)`：
- 自动根据 `Wrapper`/`ActionPrototype` 的 `name` 字段递归解码嵌套 protobuf
- 特殊处理 `GameLiveUnit`、`GameRestore`、`GameAction` 等复合类型

#### 牌谱元数据分析 `analyzeGameRecord(record_list)`

将牌谱列表元数据转为结构化 `gamedata`：

```javascript
gamedata = {
    source: 'majsoul',
    accountid: UserID,
    starttime: record.start_time,
    endtime: record.end_time,
    uuid: record.uuid,
    version: 'x.x.x',
    playerdata: [
        { id, name, rank, pt, finalpoint, deltapt }
    ],
    roomdata: {
        room: 0-5|100-104,   // 房间编号
        player: 3|4,
        round: 1|4|8,        // 1=一局, 4=东风, 8=东南
        has_ai: bool,
        init_point: 25000,
        fandian: 30000,
        dora_count: 3,
        shiduan: bool        // 食断
    }
}
```

**段位计算**：
```javascript
let idstr = String(acc.level.id);
pdata.rank = parseInt(idstr[2]) * 3 + parseInt(idstr[4]) - 3;
// 段位ID 5位数字：第3位=大段（初心/雀士/雀杰/雀豪/雀圣/魂天），第5位=小段(1-3)
// 转换为 0-17 的连续编号
```

**房间识别**：
- `config.category == 1` → 友人场
- `config.category == 2` → 段位场（`mode_id` 映射到铜/银/金/玉/王座之间）
- `config.category == 4` → 比赛场
- `mode.mode` 个位：1=一局, 2=东风, 其他=东南；十位 >= 2 表示三麻

#### 中间格式转换 `majsouldataconvert(srcdata, action)`

将雀魂原始 action 转为分析用中间格式：

| 动作类型 | 关键字段 |
|----------|----------|

**`IHule` 统计的核心指标**：
- 和牌/放铳/自摸/被自摸次数
- 和牌得点/素点（区分自摸/荣和）
- 高打分布：3900+/7700+/11600+ 占比
- 巡目：和牌/放铳时巡目
- 宝牌：表/里/赤宝牌出现次数
- 庄家相关：庄家和牌/自摸/被炸庄/放铳给庄家
- 立直相关：追立/被追立/振听立直的和牌/放铳
- 对副露者放铳：区分对手 0-4 副露
- 杠相关：本局有杠时的和牌/放铳
- 向听数：放铳时 0-6 向听分布
- 役种：56 种役在和牌/放铳/被自摸时的出现频率
- 立直宣言牌放铳

**动作分发** — `action(actstr)`：按首字母分发：
`A`=舍牌, `B`=摸牌, `C`=吃碰杠, `D`=暗杠加杠, `X`=和牌, `Y`=流局, `Z`=终局分数

#### 统计量体系

**`AnalyzeResultName`** — 从 `PAADData.json` 读取：
- `base`：基本统计量名称映射
- `result`：150+ 个结果统计量名称列表
- `resultexpr`：对应表达式列表
- `resultgroupmap` / `resultgrouporder`：分组和展示顺序
- 支持 `!startnum!endnum!` 模板语法展开（如 `FULU!1!4!` → `FULU1`~`FULU4`）

**`AnalyzeExpr`** — 表达式求值引擎：
- 四则运算（`+`, `-`, `*`, `/`）+ 括号，运算符优先级+栈算法
- 变量引用格式：
  - `BASE_xxx` — 一维基本统计量
  - `REACH_xxx_yyy` — 二维立直统计
  - `HULE_xxx_yyy` — 二维和牌统计
  - `HULEYAKU_xxx_yyy_zzz` — 三维役种统计
  - `FULU_xxx_yyy` — 二维副露统计
  - `FLOAT_xxx` — 浮点型（安定段位）
  - `RESULT_xxx` — 引用已计算的结果统计量
- `?` 通配符：`xxx?` 对所有编号求和，`xxx??` 加权求和（编号×值）

**`AnalyzeData`** — 数据存储与计算：
- `basedata`：一维 `vector<long long>`（总局数、顺位、立直、流局等）
- `reachbasedata`：二维 `[REACHBASEDATA][REACHTYPE]`（好形/愚形）
- `hulebasedata`：二维 `[HULEBASEDATA][HULEHANDTYPE]`（24 种手牌类型）
- `huleyakubasedata`：三维 `[HULEYAKUBASEDATA][YAKUDATA][HULEHANDTYPE]`
- `fulubasedata`：二维 `[FULUBASEDATA][FULUTYPE]`
- `floatdata`：一维 `vector<double>`（安定段位+置信区间）

**手牌类型分类（`gethandtype`）**：
`REACHGOOD`(好形立直), `REACHBAD`(愚形立直), `DAMA`(默听), `FULU1-4`(1-4副露), `NTDAMA`(门清未听), `NTFULU1-4`(副露未听), 以及聚合类型 `FULUPLUS1-4`, `ALLFULU`, `ALLDAMA`, `ALLREACH`, `ALL`

**`calcresult()`** 计算流程：
1. `makehanddata()` 聚合副露类型
2. 聚合立直 `ALL = GOOD + BAD`
3. 聚合副露 `FULUPLUS` 累加
4. 计算安定段位和置信区间
5. 遍历所有结果表达式求值
6. 存入 `resultjson` 用于 HTML 输出
7. 计算和牌役种和放铳役种分布

#### 筛选器 `filtercheck`

支持 `include`/`exclude` 两个方向，字段：`source`、`room`、`player`、`round`、`speed`、`timebefore`/`timeafter`、`id`、`name`、`contest_id`

## 9. 牌谱链接能提取的比赛数据字段（专题）

### 9.1 牌谱链接格式

雀魂牌谱链接格式：`paipu=UUID_xxx`，UUID 形如 `"210320-xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"`，是牌谱的唯一标识符。

### 9.2 从 UUID 能获取的数据

UUID 本身不包含任何比赛信息，只是一个标识符。所有数据都需通过 API 获取：

#### 第一层：牌谱列表元数据（`fetchGameRecordList`）

从牌谱列表中可获取：
| 字段 | 说明 |
|------|------|
| `uuid` | 牌谱 UUID |
| `start_time` / `end_time` | 对局开始/结束时间戳 |
| `config.category` | 房间类别（1=友人, 2=段位, 4=比赛） |
| `config.mode.mode` | 模式（个位：1=一局/2=东风/其他=东南；十位≥2=三麻） |
| `players[]` | 各家基础信息（nickname, avatar_id, score, gradingScore 等） |

#### 第二层：牌谱详细元数据（`fetchGameRecordsDetail`）

| 字段 | 说明 |
|------|------|
| `config.mode` | 完整模式信息 |
| `accounts[]` | 各家完整账号信息 |
| `accounts[].level.id` | 段位 ID（5 位数字：第 3 位=大段，第 5 位=小段） |
| `accounts[].level.score` | 段位分数 |
| `accounts[].chara` | 角色信息 |

#### 第三层：完整牌谱数据（`fetchGameRecord`）

获取完整的 protobuf 牌谱数据，解码后包含：

**对局信息（`head`/`gamedata`）**：
| 字段 | 说明 |
|------|------|
| `accountid` | 当前用户 ID |
| `starttime` / `endtime` | 对局时间 |
| `uuid` | 牌谱 UUID |
| `roomdata.room` | 房间等级（0=友人, 1=铜, 2=银, 3=金, 4=玉, 5=王座, 100+=比赛） |
| `roomdata.player` | 玩家数（3/4） |
| `roomdata.round` | 局制（1/4/8） |
| `roomdata.init_point` | 初始点数 |
| `roomdata.fandian` | 返点 |
| `roomdata.dora_count` | 宝牌数 |
| `roomdata.shiduan` | 食断 |
| `roomdata.has_ai` | 是否有 AI |
| `roomdata.contest_id` | 比赛场 ID |
| `playerdata[].id` | 玩家 ID |
| `playerdata[].name` | 玩家昵称 |
| `playerdata[].rank` | 段位（0-17 连续编号） |
| `playerdata[].pt` | 段位分数 |
| `playerdata[].finalpoint` | 终局点数 |
| `playerdata[].deltapt` | 段位分数变化 |

**每局数据（`record[]`）**：
| 字段 | 说明 |
|------|------|
| `round` / `chang` / `ju` | 场风/局数 |
| `ben` / `honba` | 本场数 |
| `liqibang` / `kyoutaku` | 立直棒数 |
| `scores` / `point[]` | 各家初始点数 |
| `dora` / `doras[]` | 宝牌指示牌 |
| `tiles0~tiles3` / `hand[]` | 四家配牌 |
| `left_tile_count` / `remain` | 剩余牌数 |
| `yama` | 牌山 |

**每个动作（`action[]`）—— 从 protobuf 的 `RecordXxx` 消息解码**：

| 动作类型 | 可提取字段 |
|----------|------------|
| **RecordNewRound** | 场风、局数、本场、供托、各家点数、宝牌、四家配牌、余牌数 |
| **RecordDealTile** | 摸牌者、牌面、更新后宝牌、余牌、立直信息 |
| **RecordDiscardTile** | 打牌者、牌面、是否自摸切、是否立直宣言、是否双立直、更新后宝牌 |
| **RecordChiPengGang** | 类型(吃/碰/明杠)、鸣牌者、牌组合、各牌来源 |
| **RecordAnGangAddGang** | 类型(暗杠/加杠)、杠牌者、牌面、新宝牌 |
| **RecordBaBei** | 拔北者、是否自摸切 |
| **RecordHule** | 和牌者、自摸/荣和、是否庄家、手牌、鸣牌信息、和的牌、役种列表(名+飜数)、是否役满、符数、飜数、荣和得点、自摸支付(亲/闲)、表/里宝牌、立直棒、分数变化 |
| **RecordNoTile** | 各家听牌状态和手牌、各家得失分、是否流满 |
| **RecordLiuJu** | 流局类型(九种九牌/四风连打/三家和/四家立直/四开杠) |

### 9.3 可计算的统计指标总览

从上述原始数据可计算以下统计（由 C++ 分析器完成，共 150+ 项）：

#### 总览指标
| 指标 | 公式 |
|------|------|
| 一位率 `#1R` | `#1次数 / 总场数` |
| 二位率 `#2R` | `#2次数 / 总场数` |
| 三位率 `#3R` | `#3次数 / 总场数` |
| 四位率 `#4R` | `#4次数 / 总场数` |
| 每场收支 `GAMEPROFIT` | `总得点 / 总场数` |
| 每局收支 `ROUNDPROFIT` | `总得点 / 总局数` |

#### 和牌指标
| 指标 | 公式 |
|------|------|
| 和了率 `HULER` | `和了局数 / 总局数` |
| 自摸率 `ZIMOR` | `自摸次数 / 和了次数` |
| 默听和了占比 `DAMAHULER` | `默听和了 / 总和平` |
| 副露和了占比 `FULUHULER` | `副露和了 / 总和平` |
| 平均和了打点 `HULEP` | `和了总得点 / 和了次数` |
| 平均和了素点 `HULESU` | `和了总素点 / 和了次数` |
| 满贯以上率 `HULE11600R` | `11600+次数 / 和了次数` |
| 跳满以上率 `HULE7700R` | `7700+次数 / 和了次数`（累加） |
| 3900+率 `HULE3900R` | `3900+次数 / 和了次数`（累加） |
| 平均和了巡目 `HULECC` | `和了巡目之和 / 和了次数` |
| 庄家和了率 `OYAHULER` | `庄家和了 / 坐庄次数` |

#### 放铳指标
| 指标 | 公式 |
|------|------|
| 放铳率 `CHONGR` | `放铳局数 / 总局数` |
| 平均放铳打点 `CHONGP` | `放铳总点数 / 放铳次数` |
| 放铳给默听率 `CHONGLEDAMAR` | `放铳给默听 / 放铳总数` |
| 放铳时向听数 `CHONGSHANTEN` | `各向听数次数 / 放铳总数` |
| 被自摸率 `BEIZIMOR` | `被自摸次数 / 总局数` |
| 炸庄率 `ZHAZHUANGR` | `炸庄次数 / 坐庄未和次数` |

#### 立直指标
| 指标 | 公式 |
|------|------|
| 立直率 `REACHR` | `立直次数 / 总局数` |
| 立直和了率 `HULEINREACHR` | `立直后和了 / 立直次数` |
| 立直放铳率 `CHONGINREACHR` | `立直后放铳 / 立直次数` |
| 好型立直率 `REACHGOODR` | `好型立直 / 总立直` |
| 先制立直率 `FIRSTREACHR` | `先制立直 / 总立直` |
| 平均立直巡目 `REACHCC` | `立直巡目之和 / 立直次数` |
| 立直收支 `REACHPROFIT` | `(宣言牌放铳×1000 + 总得失点) / 立直次数 - 1000` |
| 追立率/和了率/收支 | 类似先制，限定非先制立直 |
| 被追立率/和了率/收支 | 限定被追立情况 |
| 振听立直率/和了率/收支 | 限定振听立直 |

#### 副露指标（1-4 副露分别统计）
| 指标 | 公式 |
|------|------|
| N副露率 `FULURn` | `N副露次数 / 总局数` |
| N副露和了率 `HULEINFULURn` | `N副露和了 / N副露次数` |
| N副露放铳率 `CHONGINFULURn` | `N副露放铳 / N副露次数` |
| N副露和了打点 `FULUHULEPOINTn` | `N副露和了总点 / N副露和了数` |
| N副露收支 `FULUPROFITn` | `N副露总得失 / N副露次数` |

#### 流局指标
| 指标 | 公式 |
|------|------|
| 流局率 `LIUJUR` | `普通流局数 / 总局数` |
| 流局听牌率 `LIUJUTENPAIR` | `听牌次数 / 流局次数` |
| 流局收支 `LIUJUPROFIT` | `(听牌收入 + 未听支出) / 流局次数` |

#### 杠牌指标
| 指标 | 公式 |
|------|------|
| 总杠率 `GANGR` | `(明杠 + 加杠 + 暗杠) / 总局数` |
| 杠后和了率 `GANGHULER` | `杠后和了 / 杠次数` |
| 杠后放铳率 `GANGCHONGR` | `杠后放铳 / 杠次数` |
| 立直后暗杠率 `REACHANGANGR` | `立直后暗杠 / 立直次数` |

#### All Last 指标
| 指标 | 公式 |
|------|------|
| AL 保一率 | `AL保一次数 / AL一位次数` |
| AL 逆一率 | `AL逆一次数 / AL非一位次数` |
| AL 避四率 | `AL避四次数 / AL四位次数` |

#### 安定段位
| 指标 | 公式 |
|------|------|
| 安定段位 `SR` | `(前三位总顺位pt + 总马点pt) / 四位次数` → 段位映射 |
| 95% 置信区间 | 蒙特卡洛 1000 次模拟 + t 分布，放大 1.4 倍 |

#### 役种分布
- 56 种役在和牌时/放铳时/被自摸时的出现频率
- DORA/URA/AKA 统计翻数而非次数
